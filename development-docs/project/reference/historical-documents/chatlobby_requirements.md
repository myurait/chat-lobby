# ChatLobby 要件定義書（初版）

## 1. 本書の目的

本書は、ユーザーが現在利用している ChatGPT / Claude Code / Codex 周辺の分断を解消し、
「スマホ起点の会話から、調査・仕様化・実装依頼・修正依頼・状況確認までを一貫して行える統合運用系」を構築するための要件を整理したものである。

本書は、現時点での要求理解を固定するための初版であり、製品選定や実装計画の前提資料として利用する。

---

## 2. 要求の主題

本要求の主題は、個別のAI製品を単独で便利に使うことではない。
複数のAIエージェントやモデルを、1つのチャット基盤の中で、ゆるやかな会話文脈を保ちながら運用できるようにすることである。

求めているものは、厳密なプロジェクト管理ツールではなく、以下の性質を持つ統合チャット基盤である。

- ラフな会話から仕事を依頼できる
- 会話の流れの中で、調査・仕様化・実装・修正・仕様差し戻しが自然に行える
- ユーザーが毎回 Claude / Codex / ChatGPT のどれを使うか判断しなくてよい
- 各エージェントの作業状況や成果が、前面のチャット上から把握できる
- ドキュメント、コード、過去プロジェクトの知見が横断的に参照される
- プロジェクトごとの文脈はある程度分離されつつ、必要に応じて越境参照できる

---

## 3. 現在困っていること

### 3.1 ドキュメントの孤立

ChatGPT プロジェクト内で作成した資料が、ダウンロードしなければ外部で利用できない。
そのため、ChatGPT 上で整理した仕様や議論を、そのまま Claude Code や Codex の作業入力として使いにくい。

### 3.2 スマホからの作業統制不足

スマホから Claude 側へ指示を飛ばせず、また作業中の内容や状態を確認しづらい。
モバイル起点での統合運用が成立していない。

### 3.3 モデル選択の手動判断

Claude と Codex のどちらに依頼すべきかを、その都度ユーザーが判断しなければならない。
この判断コストが高く、会話から作業への移行を阻害している。

### 3.4 フック・ナレッジの分断

Claude / Codex のフック、ルール、知識、補助情報がそれぞれ閉じており、共有されない。
そのため、片方で得た知見や運用規約が他方へ自然に反映されない。

### 3.5 手元環境とチャット環境の断絶

ローカルのコードベースやドキュメントに変更が入っても、スマホで使っている ChatGPT からその状態を参照できない。
結果として、ChatGPT 側の議論と実際の実装状態が乖離する。

### 3.6 プロジェクト横断知識の再利用不足

ChatGPT プロジェクトのように、「今のプロジェクトに集中しつつ、必要なら過去のプロジェクトの知見を曖昧に参照する」体験が不足している。
例えば「前のプロジェクトで使った規約を流用して」といった指示が通りにくい。

---

## 4. 理想像

### 4.1 基本イメージ

理想は、PdM 視点でチャット上から開発チームに指令できる統合基盤である。
前面には応対する開発リーダー相当のエージェントが存在し、その背後で複数のエージェントが並列に動く。

ユーザーは基本的に前面の会話相手とだけ話せばよく、
必要に応じて裏側で他のエージェントが起動・分担・調査・実装・修正を行う。

### 4.2 理想の会話体験

- スマホから自然言語で話しかけられる
- 調査相談から仕様化に移行できる
- 仕様化した内容を、そのまま実装担当エージェントへ渡せる
- 実装結果を見て仕様の修正が必要なら、再び仕様担当へ差し戻せる
- その修正版仕様が、以後の主要ナレッジとして共有される
- 会話の流れの中で、複数のエージェントが自律的に役割分担する

### 4.3 理想の知識共有

- ドキュメントはスマホから閲覧できる
- 可能であればスマホから編集もできる
- Codex / Claude / 前面エージェントが同じドキュメントを参照できる
- 同じドキュメントを前提に議論できる
- 他プロジェクトの知見も参照可能である
- ただし、現在集中しているプロジェクトの文脈は自然に優先される

---

## 5. 想定ユースケース

### 5.1 会話から仕様策定まで

ユーザーがチャット上で調査や雑談を行う。
その流れの中で、新しいサービス案や機能案を深掘りし、仕様書相当の文書をまとめる。

### 5.2 仕様から実装依頼まで

まとまった仕様をもとに、ユーザーが「そろそろコーディングに移る」と判断する。
前面エージェントは、適切な実装担当エージェントへ作業を振り分け、プロトタイプ開発を開始させる。

### 5.3 複数実装担当による並列修正

プロトタイプ完成後、要件の取り込み漏れや改善点が見つかる。
前面エージェントは、仕様・コード・進行中タスクを踏まえて、複数の実装担当へ別々の修正作業を振り分ける。

### 5.4 実装から仕様差し戻しまで

実装を見た結果、「そもそも仕様の書き方がよくない」と判断される場合がある。
その際、前面エージェントは現在のコードベース、作業内容、既存仕様を統合し、新しい仕様書を再生成・更新する。

### 5.5 過去プロジェクト知見の再利用

ユーザーが「前のプロジェクトで使った規約や進め方を流用したい」と指示する。
前面エージェントは、関連する過去プロジェクトの知見を参照し、現在のプロジェクト文脈に応じて再利用する。

---

## 6. スコープ定義

### 6.1 本要件で対象とするもの

- AIチャット統合基盤の会話体験
- 複数エージェントへの自動振り分け
- ドキュメント共有
- 作業状況の可視化
- プロジェクト文脈管理
- プロジェクト横断知識参照
- スマホからの操作性

### 6.2 本要件で対象外とするもの

以下は現時点では必須とはしない。

- ChatGPT / Claude / Codex の3製品すべてを、対等なメンバーとして必ず統合すること
- 既存公式アプリのみで完結させること
- 完全自動で全工程を無人化すること
- 厳密なプロジェクト管理機能やチケット管理を主目的にすること
- 会社向けの厳格な権限制御や監査要件を初期段階から満たすこと

---

## 7. 必須要件

### 7.1 会話基盤要件

#### 7.1.1 前面エージェントの存在
ユーザーは基本的に1つの前面エージェントと会話すればよいこと。
前面エージェントが、裏側の複数エージェントとの仲介役となること。

#### 7.1.2 ラフな会話からの作業移行
雑談、調査相談、仕様相談、実装依頼、修正依頼が、同じ会話圏で連続して行えること。

#### 7.1.3 モデル選択の隠蔽
Claude / Codex などの使い分けを、原則としてユーザーが明示判断しなくてよいこと。
利用モデルや担当エージェントの選定は、基盤側が内部で処理すること。

### 7.2 スマホ利用要件

#### 7.2.1 モバイルからの指示
スマホから、前面エージェントへ自然言語で指示できること。

#### 7.2.2 モバイルからの状態確認
スマホから、各エージェントの進行状況、作業中内容、完了状況を確認できること。

#### 7.2.3 モバイルからの文書参照
スマホから、現在の仕様書や関連ドキュメントを閲覧できること。

### 7.3 ドキュメント要件

#### 7.3.1 共有正本の存在
仕様書、設計メモ、作業指示書などの主要文書には、各エージェントが共通で参照する共有正本が存在すること。

#### 7.3.2 チャットと文書の接続
会話の中で確定した内容を、共有正本へ反映できること。
また、共有正本の内容を会話に再投入して議論できること。

#### 7.3.3 実装修正との往復
コードベース側の変更結果を踏まえ、仕様書を更新できること。
更新後の仕様書は、以後の主要ナレッジとして再利用されること。

### 7.4 エージェント連携要件

#### 7.4.1 並列起動
必要に応じて、背後で複数エージェントが並列起動できること。

#### 7.4.2 自動振り分け
作業内容に応じて、どの実装担当エージェントを使うか自動的に選択できること。

#### 7.4.3 作業結果の統合
複数エージェントの作業結果を、前面エージェントが統合し、ユーザーに提示できること。

### 7.5 知識管理要件

#### 7.5.1 プロジェクト文脈の緩い分離
プロジェクトごとに、主要な文脈や優先ナレッジは分離されること。

#### 7.5.2 プロジェクト横断参照
必要に応じて、他プロジェクトの知見、規約、設計パターンを参照できること。

#### 7.5.3 再利用指示の理解
「前のプロジェクトで使った規約を使いまわして」といった自然言語指示が解釈可能であること。

---

## 8. 準必須要件

以下は重要だが、初期構築時には段階導入でもよい。

### 8.1 スマホからの文書編集
閲覧だけでなく、軽度な修正やコメント追記がスマホから行えること。

### 8.2 コード状態の参照
前面エージェントが、ローカルまたは共有リポジトリ上の最新コード状態を参照可能であること。

### 8.3 作業履歴の蓄積
誰がどの文脈で何を変更したか、作業履歴や判断履歴をある程度追跡できること。

### 8.4 ナレッジの自動昇格
会話中に得られた重要知見を、自動または半自動で共有ナレッジへ昇格できること。

---

## 9. 非必須要件

以下は現時点では望ましいが must ではない。

- ChatGPT / Claude / Codex の3者すべてを必須構成要素とすること
- 既存の各公式製品そのものを中心 UI として残すこと
- 単一製品のみで全要件を満たすこと
- 完全自律で人間確認なしに全てを進行させること
- 厳密なリアルタイム共同編集機能
- 厳密な企業向け権限分離や監査証跡

---

## 10. 制約に関する理解

### 10.1 実現手段の自由度
既製品のみで成立しなくてもよい。
必要であれば、オーケストレーション層、メモリ管理層、統合チャット層をスクラッチ開発してよい。

### 10.2 製品構成の柔軟性
ChatGPT が統合対象から外れてもよい。
例えば、Codex / Claude の統合だけで要件の大半を満たせるなら、その構成も候補として許容される。

### 10.3 会話中心設計
中心に置くべきなのは、プロジェクト管理機能ではなくチャット体験である。
プロジェクトやタスクは、会話を支える裏側の構造として存在すればよい。

---

## 11. 本書時点での要求理解の要約

本要求は、以下の一文に集約される。

「スマホから自然に話しかけるだけで、前面のAIが応対し、背後の実装エージェント群へ適切に作業を振り分け、共通ドキュメントと共通ナレッジを参照しながら、調査・仕様・実装・修正・仕様更新を往復できる統合チャット基盤が欲しい」

---

# ChatLobby 理想体験仕様書

## 1. この文書の目的

この文書は、ChatLobby の roadmap2 以降を体験設計主導で進めるための正本である。

本プロジェクトの課題は、単に複数の AI ツールを 1 つの画面から呼べるようにすることではない。
ユーザーが感じている本当の不満は、以下にある。

- 会話、仕様、コード、タスク、判断履歴が複数の AI とツールに散逸している
- どの AI に何を頼むべきかを毎回自分で考えなければならない
- 同じ文脈を繰り返し説明し直さなければならない
- スマホから気軽に指示したいのに、実際には入口が多く、状態の把握も困難である
- 過去の重要な会話が埋もれ、「そういえばあの話」が自然につながらない
- 壁打ちを始めてから project として昇格させるまでに、文書形式や構成を考える負担が発生する
- 新しい会話か、前の会話の続きかを自分で説明するコストが惜しい

ChatLobby が目指すべきものは、これらを局所的に改善する便利ツールではなく、
「前面 AI に話しかければ、背後の複数ワーカー、知識、文書、履歴が適切に接続された状態で開発が進む体験」である。

## 2. 目指すプロダクトの一言定義

ChatLobby は、前面 AI を入口として、会話、記憶、文書、タスク、配下ワーカーを統合し、
ユーザーが文脈再説明やツール切替なしに継続開発を進められる self-hosted 開発 frontdoor である。

## 3. 想定ユーザー像

### 3.1 中核ユーザー

- 個人または少人数で複数プロジェクトを並行する開発者
- AI を仕様整理、実装、調査、設計補助に積極的に活用している人
- スマホとデスクトップをまたいで断続的に作業する人
- 会話をその場限りの壁打ちではなく、継続的な開発資産として扱いたい人

### 3.2 現在の不満

- ChatGPT、Claude Code、Codex、ローカルツールが分断されている
- どこに何を残したかを思い出すコストが高い
- 会話が増えるほど、過去の議論を再利用しにくくなる
- 「今どの AI が何をしているか」が見えない
- ちょっとした指示でも、プロジェクト、スレッド、モデル、作業場所を意識しなければならない
- 壁打ちを project に昇格させるとき、文書ルールや配置を考える時間が惜しい
- 「これは新しい会話ですか？」のようなメタ確認に手数を使いたくない

## 4. 理想体験の要約

理想の ChatLobby では、ユーザーは基本的に 1 人の前面 AI とだけ会話すればよい。

ユーザーは「この件を調べて」「この仕様をまとめて」「前の続きだけど」「あの件のバグを直して」と自然言語で話しかける。
前面 AI は、その発話がどのプロジェクト、どのスレッド、どの継続タスクに属するかを理解し、必要な文脈を自動で集め、適切な配下ワーカーに作業を割り当てる。

また、壁打ちの段階から project へ昇格させるまでの導線は、ユーザーにとってほとんど見えない。
ユーザーは doc の置き場、テンプレート、命名規約、文書種別を先に考えさせられず、会話の中で本質的な設計や判断へ素早く入れる。

その結果、ユーザーは以下をしなくてよくなる。

- どの AI に頼むべきかを毎回判断する
- 過去の会話を探し回る
- 同じ背景説明を何度も書き直す
- 文書・仕様・タスクの保存先を毎回意識する
- 今何が動いていて何が止まっているかを手作業で追う
- これは新しい話か、前の続きかを逐一明示する
- 壁打ちを project に育てる前段として、文書体裁を先に整える

## 5. 理想体験の中核原則

### 5.1 入口は 1 つである

ユーザーが意識する入口は、基本的に前面 AI だけである。
背後に Claude Code、Codex、knowledge、将来の外部エージェントがいても、通常利用時にそれらを直接選び分ける必要はない。
必要なときだけ override できればよい。

### 5.2 文脈再説明を最小化する

ユーザーは、前回の会話や関連文書を毎回貼り直すべきではない。
前面 AI は、会話履歴、プロジェクト文脈、関連文書、過去タスク、決定記録を参照し、継続文脈を復元する責務を持つ。

### 5.3 継続会話が自然に成立する

ユーザーが「そういえばあの話だけど」「前に詰めていた件だけど」と言ったとき、前面 AI は高い確率で適切な文脈候補を特定できなければならない。
その際、ユーザーへ「これは新しい会話ですか、既存の会話ですか」と管理的な確認を押しつけるべきではない。
まず AI 側が文脈を推定し、必要がある場合だけ短く補正可能な形で会話を返すべきである。

### 5.4 誤判定しても、自由に文脈を切り直せる

自動文脈判定は便利であるが、誤接続は必ず起こる。
したがって理想状態では、AI が誤って既存 project / thread に接続しても、その後の対話で自然に修正できる必要がある。

理想的な流れの例:
- AI「このプロジェクトの話ですね。前回の続きとして扱います」
- User「勘違いさせた。別の話だ」
- AI「失礼しました。新しい会話スレッドに戻します」

重要なのは、誤接続が起きないことではなく、誤接続が起きてもユーザーが窮屈さを感じずに再ルーティングできることである。

### 5.5 配下ワーカーの存在は感じさせても、負担は感じさせない

ユーザーは「何かが動いている」ことは把握できるべきだが、「自分で全部交通整理しなければならない」と感じてはならない。
前面 AI が監督者として整理し、必要な可視化だけを返す。

### 5.6 知識は会話の彼方へ消えず、必要な形で昇格される

その場の会話に価値があっても、放置すれば散逸する。
重要な決定、仕様、TODO、知見は、適切なタイミングで Git 正本や共有知識へ昇格されるべきである。
ただし、すべてを即時に固定するのではなく、一時記憶と正本化を分けて運用する。

### 5.7 壁打ちから project への昇格は frictionless である

ユーザーは、曖昧な相談、構想段階の壁打ち、雑多な思考断片から出発することが多い。
理想状態では、その雑さを前面 AI が吸収し、必要な thread、project、文書、初期 backlog、設計文書へ滑らかに昇格させる。

ここで重要なのは、ユーザーが以下に悩まされないことである。
- この話はまだメモ段階なのか、もう project なのか
- どの文書から切るべきか
- spec と ADR と feature backlog のどれに書くべきか
- docs の体裁をどう整えるべきか

これらは AI 側が吸収し、ユーザーは本質的な設計・判断・優先度付けに早く到達できるべきである。

### 5.8 スマホからでも開発の流れが切れない

ChatLobby はスマホ起点で使える必要がある。
スマホから相談し、指示し、進捗を見て、必要なら修正指示を出し、後でデスクトップに戻っても同じ流れを継続できることが重要である。

## 6. 理想体験が成立している状態

次の状態が、ChatLobby の理想体験が成立している基準である。

1. 新しい相談を投げると、前面 AI が必要な文脈を自動で集めて整理し、適切なワーカーに回す。
2. 既存の話題の続きであれば、ユーザーが thread や project を明示しなくても、前面 AI が候補を理解する。
3. 誤った thread / project に接続しても、会話の中で自然に切り替えられる。
4. 壁打ちから始めても、project として扱うべき段階になったら、前面 AI が必要文書や構造を吸収したまま昇格させる。
5. 配下ワーカーが複数動いていても、ユーザーは前面 AI に「今どうなってる？」と聞けば整理された状況を受け取れる。
6. 会話で確定した内容が、仕様、ADR、worklog、knowledge として適切に昇格される。
7. 過去の別プロジェクトの知見を、現在の作業に必要な範囲で再利用できる。
8. 必要なときだけモデル、sandbox、approval などの実行条件を調整できる。
9. ツール切替や履歴探索や文書体裁調整が主作業にならず、開発判断そのものに集中できる。

## 7. ユーザーが得たい感覚

ChatLobby が成功したとき、ユーザーは次のように感じる。

- 「前の会話や過去プロジェクトをちゃんと覚えてくれている」
- 「どの AI に頼むかを考えなくてよい」
- 「途中で席を外しても、戻れば流れが再開できる」
- 「相談、仕様整理、実装依頼、状況確認が別々の作業に見えない」
- 「AI を個別に使い分けるより、1 つのチームとして扱えている」
- 「壁打ちがそのまま project に育ち、体裁調整に時間を奪われない」
- 「会話の置き場を管理するのではなく、会話そのものに集中できる」

## 8. 代表的な利用シナリオ

### 8.1 スマホから新しい相談を始める

ユーザーは通勤中や外出先で、スマホから前面 AI に話しかける。

例:
- 「Open WebUI の現状 UI だと厳しい。ChatLobby としての見た目を再検討したい」
- 「昨日の会話を踏まえて、front agent の責務を整理して」

理想状態では、前面 AI は以下を行う。

- 発話がどの project / thread に近いかを判定する
- 関連する仕様、設計、過去会話を拾う
- いま新規 thread を立てるべきか、既存へ接続するべきかを判断する
- 相談として整理するか、設計タスクへ分解するかを決める
- 必要なら配下ワーカーや知識検索へ振る

ユーザーは、保存先や対象モデルを考えず、会話に集中できる。

### 8.2 「そういえばあの話だけど」で前の議論を再開する

ユーザーは、詳細を覚えていないまま次のように話しかける。

- 「そういえばあの話だけど、thread 自動振り分けはやっぱり重要だよね」
- 「前に deferred に落とした件、もう一回見直したい」

理想状態では、前面 AI は以下を行う。

- 過去の会話、feature、roadmap、設計文書から候補を引く
- もっとも有力な文脈を推定する
- 必要であれば 1 回の軽い補正で文脈を確定する
- 以前どこまで決まっていたか、何が未確定かを整理して返す

ここで重要なのは、単なる全文検索ではなく、「前の流れを把握している会話相手」と感じられることである。

### 8.3 文脈誤判定から自然に戻す

ユーザーの発話が曖昧だったため、前面 AI が誤って既存 project へ接続することはあり得る。

例:
- AI「このプロジェクトの話ですね。前回の thread routing の続きとして扱います」
- User「いや、別件。新しい話だよ」
- AI「失礼しました。新しいスレッドとして切り替えます。関連候補は外します」

理想状態では、ここで以下が成立する。

- AI は訂正を受けたら即座に現在の会話コンテキストを再計算する
- 誤って引いていた project / thread / task を会話の主文脈から外す
- 必要なら新しい thread / project 候補を組み直す
- ユーザーに thread 管理操作を強制しない

### 8.4 壁打ちから project に昇格する

ユーザーは最初から整理された要件を書けるとは限らない。

例:
- 「なんとなく思ったが、ChatLobby の会話管理って根本的に弱い気がする」
- 「この辺を一度ちゃんと整理して、次の roadmap にしたい」

理想状態では、前面 AI は以下を行う。

- 会話から論点、課題、理想体験、設計対象を抽出する
- project として持つべき文書や backlog の骨組みを裏側で用意する
- 必要なら review、design、roadmap、feature epic へ昇格候補を整理する
- ただし、ユーザーに文書種別選択や配置判断を先に要求しない

ユーザーは、ドキュメント運用の前準備ではなく、体験設計と判断そのものに素早く入れる。

### 8.5 前面 AI が監督者として配下ワーカーを動かす

ユーザーは次のように依頼する。

- 「front agent の責務を設計して、それに必要なドキュメント案も出して」
- 「この feature backlog を理想体験ベースで再編したい」

理想状態では、前面 AI は以下を行う。

- 発話を理解し、調査、文書整理、実装、レビューのどれが必要かを判断する
- 配下ワーカーに作業を割り当てる
- 結果をそのまま投げ返さず、統合・要約・整理したうえで返す
- 追加で必要な設計文書や ADR を指示する
- 進行中の作業を後から照会できるよう状態を残す

### 8.6 仕様と実装の往復が会話の中で閉じる

ユーザーは、会話で仕様を詰め、そのまま実装依頼し、結果を見てまた仕様を修正したい。

理想状態では、ChatLobby は以下を実現する。

- 会話から仕様案を生成する
- 必要に応じて Git 正本へ昇格する
- 実装ワーカーへ依頼する
- 実装結果や差分を受けて、仕様や判断記録の更新候補を返す
- 必要なら次のタスクへ自動で接続する

### 8.7 進行状況を自然言語で確認する

ユーザーは「今どうなってる？」「何が止まってる？」「承認が必要なものある？」と聞くだけでよい。

理想状態では、前面 AI は以下を返す。

- いま動いているタスク
- 各タスクの担当ワーカー
- 現在ステップ
- どこで止まっているか
- ユーザー判断待ちかどうか
- 重要な変更ファイルや結果概要

ここで必要なのは raw event の羅列ではなく、「人間が次に何をすればよいかが分かる整理」である。

## 9. 前面 AI が担うべき責務

前面 AI は単なる chat UI のラッパではない。
次の責務を明示的に持つ必要がある。

### 9.1 会話理解

- ユーザー発話の意図分類
- 新規相談、継続相談、実装依頼、レビュー依頼、状況確認、知識探索の判定
- 会話開始直後に、既存文脈へ接続すべきか新規文脈を立てるべきかの推定

### 9.2 文脈復元

- 関連 thread、document、task、project の推定
- 必要な会話履歴と正本知識の収集
- 曖昧参照の解決
- 誤接続時の再計算

### 9.3 タスク分解

- 大きな依頼を、調査、設計、実装、レビューなどのサブタスクへ分解する
- どの作業をどのワーカーへ渡すか決める

### 9.4 実行統制

- ワーカー選定
- 実行条件の決定
- 必要に応じた override 受理
- 状態の追跡

### 9.5 結果統合

- 配下ワーカーの生出力をそのまま返さず、会話として再構成する
- 矛盾、抜け漏れ、次アクションを整理する

### 9.6 知識昇格判断

- どの内容を specs / decisions / worklog / knowledge に昇格するべきか判断する
- 会話資産の散逸を抑える

### 9.7 project 昇格支援

- 壁打ち段階の会話から project 化に必要な骨組みを抽出する
- 必要文書、初期 epic、thread、context を裏側で整える
- ユーザーに文書運用の手間を先に負わせない

## 10. 記憶と文脈の要件

理想体験の核は、前面 AI が「覚えていること」にある。
ただし、何でも一様に記憶するのではなく、記憶層を分ける必要がある。

### 10.1 一時記憶

- 直近会話の流れ
- 暫定的な論点
- まだ正本化していない判断候補
- 作業中タスクの状態
- 一時的に誤接続された文脈と、その訂正履歴

### 10.2 構造化記憶

- thread と task の対応
- project ごとの主要論点
- feature、roadmap、ADR、仕様書とのリンク
- 関連会話候補
- project 昇格前後のつながり

### 10.3 永続知識

- Git 正本に昇格した specs / decisions / worklog / knowledge
- 再利用価値の高い設計方針、教訓、規約

### 10.4 必要な記憶機能

- 類似会話検出
- 既存 thread への接続提案または自動接続
- project 境界を跨ぐ関連知識探索
- imported conversation の再マッピング
- 記憶対象の昇格・要約・整理
- 文脈誤判定後の再ルーティング

## 11. thread と project の理想的な扱い

ユーザーは thread 管理を主作業にしたくない。
しかし、完全な自動接続だけでは誤接続が起きる。

よって理想状態では次が必要である。

- システム側が有力な thread / project 候補を判断する
- 高確度なら自動接続する
- 低確度なら短い補正可能な会話で決める
- project 固有文脈と全体共通知識を分けて保持する
- 過去プロジェクトから現在に再利用する際は、その根拠が見える
- 誤接続した場合、現在の会話を別 context に切り直せる
- thread への接続判断と project 昇格判断を別問題として扱う

## 12. 配下ワーカー可視化の理想状態

可視化は必要だが、ノイズの多い raw thinking の垂れ流しは主目的ではない。

理想状態では、ユーザーが通常見るべきなのは以下である。

- 誰が何をしているか
- 今どこで止まっているか
- 何が変わったか
- ユーザー判断が必要か
- 次に何をすべきか

thinking の別パネル表示は、必要なら上級者向け補助として存在し得る。
ただし、これは中核体験ではなく、監督性・透明性を補助する二次要素として扱う。

## 13. 文書と知識の理想ループ

ChatLobby は、会話をその場のノイズとして終わらせず、必要な部分を文書化し、将来の作業へ戻せるべきである。

理想ループは以下である。

1. 会話で相談・整理する
2. 必要に応じて feature / spec / ADR / worklog として昇格する
3. 実装ワーカーがそれを参照して作業する
4. 結果を受けて文書更新候補が返る
5. 更新後の文書が次回以降の共通知識になる

重要なのは、ユーザーが保存先や文書種類を毎回考えなくてよいことである。
前面 AI が「これは仕様に上げるべき」「これは判断記録として残すべき」と整理できることが望ましい。

## 14. project 昇格の理想ループ

壁打ちから project に昇格する場面では、次の流れが frictionless に成立する必要がある。

1. ユーザーが曖昧な問題意識や構想を会話で出す
2. 前面 AI が論点、課題、理想状態、設計対象を抽出する
3. 会話が継続案件だと判断したら、必要な thread / project 文脈を裏側で整える
4. 必要文書や feature epic の初期骨組みを AI 側で提案または生成する
5. ユーザーは project 体裁ではなく、本質的な設計判断へ移る

失敗してはならないのは、以下である。

- project 昇格のたびに、ユーザーが文書運用を手作業で再設計すること
- project 化が documentation ceremony になり、本質的な議論開始が遅れること
- 昇格後に、元の壁打ち会話とのつながりが消えること

## 15. 実行制御の理想状態

通常時、ユーザーは実行条件を細かく意識しない。
しかし、必要な場面では制御できるべきである。

理想状態では以下が両立する。

- デフォルトでは前面 AI が最適な worker / model / runtime を選ぶ
- 必要ならユーザーが sandbox、approval、model などを明示的に override できる
- override は例外的行為であり、通常体験を壊さない

## 16. 将来拡張として価値が高い要素

以下は理想体験に寄与するが、基礎体験成立後でもよい。

- ChatGPT 会話資産のインポート
- 音声入力
- ローカル LLM やその他エージェントへの adapter 拡張
- thinking の別パネル表示
- MCP server 化

## 17. 結論

ChatLobby の理想体験は、「複数の AI を前面 UI から呼べること」では足りない。
本質は、前面 AI が会話と開発の監督者となり、ユーザーから次の負担を奪うことにある。

- 文脈の再説明
- ツール切替
- 会話の置き場管理
- 文書体裁の前準備
- 新規 / 継続会話の判定
- 誤判定後の修正コスト

この負担が消えて初めて、ユーザーは壁打ちから project へ速やかに移行し、体験設計や仕様判断という本質的な開発サイクルに集中できる。
